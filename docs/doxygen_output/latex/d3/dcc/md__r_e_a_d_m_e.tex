{\bfseries{Compact\+Star}} is a high-\/performance, modular C++ framework for modeling the microphysics, structure, and evolution of compact stars‚Äîneutron stars, hybrid stars, and dark‚Äìvisible admixed configurations.

Originally designed for dense-\/matter astrophysics research, the codebase has expanded into a full computational platform with\+:

\item thermal + chemical evolution engines,
\item baryon-\/number‚Äìviolating (BNV) processes and exotic heating channels,

\item sequences, contours, and bulk EOS analysis,
\item a modernized Core module providing math, physics, datasets, file I/O, and analysis utilities.
\end{DoxyItemize}

Compact\+Star is written in {\bfseries{C++17}}, designed for {\bfseries{flexibility, reproducibility, and extensibility}}, and integrates cleanly into HPC workflows.

\hypertarget{md__r_e_a_d_m_e_autotoc_md45}{}\doxysection{State Interface Architecture}\label{md__r_e_a_d_m_e_autotoc_md45}
Compact\+Star‚Äôs evolution engine treats each physical subsystem ~\newline
 (Spin, Thermal, Chem, BNV, ‚Ä¶) as a {\bfseries{state block}} derived from a single abstract base class {\ttfamily State}. This base class defines a small, uniform interface\+:


\begin{DoxyItemize}
\item {\ttfamily Size()}
\item {\ttfamily Data()}
\item {\ttfamily \mbox{\hyperlink{_chem_state_8hpp_a3f6089e2db736965f021e2165ae39933}{Pack\+To()}}}
\item {\ttfamily \mbox{\hyperlink{_chem_state_8hpp_af4c43758909d4e294d63b7d067eae761}{Unpack\+From()}}}
\item {\ttfamily Resize()}
\item {\ttfamily \mbox{\hyperlink{_chem_state_8hpp_ac2fdf117e3886bbce4baa56a77d3e9cc}{Clear()}}}
\end{DoxyItemize}

but {\bfseries{it intentionally does not define storage}}. ~\newline
 Each derived state type owns its own internal representation.

\DoxyHorRuler{0}
\hypertarget{md__r_e_a_d_m_e_autotoc_md47}{}\doxysubsection{Why the base class does $<$em$>$not$<$/em$>$ own $<$tt$>$values\+\_\+$<$/tt$>$}\label{md__r_e_a_d_m_e_autotoc_md47}
Different state types have very different physical and numerical needs\+:


\begin{DoxyItemize}
\item {\bfseries{Spin\+State}} stores Œ©-\/like variables in a simple vector
\item {\bfseries{Thermal\+State}} may evolve one or many temperature components
\item {\bfseries{Chem\+State}} evolves a vector of Œ∑ chemical imbalances
\item {\bfseries{BNVState}} may evolve anything from 1 to many exotic parameters
\item Future modules may require 2D grids, GPU buffers, or nested structures
\end{DoxyItemize}

If the base class forced a single storage container (e.\+g. {\ttfamily std\+::vector\texorpdfstring{$<$}{<}double\texorpdfstring{$>$}{>}}), it would prevent advanced future models or force awkward workarounds.

Leaving storage fully up to the derived class gives\+:


\begin{DoxyItemize}
\item full flexibility for arbitrary physics models, present and future
\item decoupling between ‚Äúhow data is stored‚Äù and ‚Äúhow evolution operates‚Äù
\item identical external interface for all states
\item zero-\/cost abstraction\+: the solver only sees contiguous segments via {\ttfamily Data()}
\end{DoxyItemize}

Every state remains free to choose {\bfseries{any internal representation}}, as long as it can supply a contiguous DOF view to the evolution engine.
